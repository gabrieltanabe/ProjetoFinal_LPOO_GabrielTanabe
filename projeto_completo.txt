## 1. DescriĂ§ĂŁo e DelimitaĂ§ĂŁo do Escopo
- **PropĂłsito:** Automatizar e gerenciar o processo de venda de ingressos de um cinema e seu catĂĄlogo de exibiĂ§ĂŁo.
- **Contexto de Uso:** O sistema operarĂĄ em totens de autoatendimento ou guichĂȘs do cinema. O ambiente Ă© dividido entre a ĂĄrea do usuĂĄrio final (cliente comprando o ingresso) e a ĂĄrea restrita para os administradores.

- **Problema Resolvido:** Elimina o controle manual de lotaĂ§ĂŁo de salas, previne o *overbooking* (venda duplicada de um mesmo assento) e padroniza a persistĂȘncia financeira e gerencial das vendas diĂĄrias.

### Diagrama de Casos de Uso (PlantUML)



### DocumentaĂ§ĂŁo dos Casos de Uso

**UC01 - Gerenciar CatĂĄlogo**
- **Atores:** Administrador
- **PrĂ©-condiĂ§Ă”es:** O sistema e o banco de dados devem estar conectados.
- **Fluxo Principal:** 1. O Admin seleciona a aba de Filmes, Salas ou SessĂ”es.
  2. Preenche os dados solicitados no formulĂĄrio.
  3. Clica no botĂŁo de aĂ§ĂŁo (Salvar/Atualizar/Excluir).
  4. O sistema valida as entradas (Controllers) e atualiza o banco de dados (DAO).
- **Fluxo Alternativo:** Se um dado for invĂĄlido (ex: duraĂ§ĂŁo negativa), o sistema exibe um Pop-up de erro e aborta.

**UC02 - Comprar Ingresso**
- **Atores:** Cliente
- **PrĂ©-condiĂ§Ă”es:** Devem existir sessĂ”es cadastradas como "DisponĂ­veis".
- **Fluxo Principal:** 1. O Cliente seleciona uma sessĂŁo na listagem (Treeview).
  2. Insere o nĂșmero do assento e seleciona o Tipo de Ingresso.
  3. Seleciona a Forma de Pagamento e clica em Confirmar.
  4. O sistema gera a instĂąncia de Ingresso.
  5. O sistema chama o UC03 (Processar Pagamento).
  6. A Venda Ă© salva e o assento torna-se indisponĂ­vel.

**UC03 - Processar Pagamento**
- **Atores:** Cliente (Sistema)
- **PrĂ©-condiĂ§Ă”es:** O Ingresso deve ter sido instanciado e seu valor calculado.
- **Fluxo Principal:** O sistema repassa o valor Ă  classe `Strategy` selecionada (PIX, CartĂŁo, Dinheiro), valida a aprovaĂ§ĂŁo e consolida o recibo.

Este projeto implementa um sistema de bilhetagem de cinema desenvolvido em Python, com interface grĂĄfica (GUI) e persistĂȘncia em banco de dados. O sistema foi reestruturado a partir do repositĂłrio de Tecnologia de OrientaĂ§ĂŁo a Objetos  para aplicar a arquitetura MVC (Model-View-Controller) e o padrĂŁo DAO (Data Access Object). ## DocumentaĂ§ĂŁo e Diagramas UML



## Principais Componentes e Arquitetura
O sistema foi modularizado separando as responsabilidades:

### Model (Regras de NegĂłcio)
- **Filme, Sala e Sessao** formam o nĂșcleo.
- **Ingresso** atua como base abstrata para **IngressoComum**, **IngressoMeia** e **IngressoVip**.
- O mĂłdulo de pagamento utiliza o padrĂŁo Strategy, contendo as classes concretas **PagamentoDinheiro**, **PagamentoPix** e **PagamentoCartao**.
- **Venda** integra ingresso e pagamento, consolidando a transaĂ§ĂŁo.

### DAO (PersistĂȘncia)
- **GenericDAO** Ă© a classe abstrata que define as operaĂ§Ă”es base do CRUD (salvar, listar, remover, atualizar, buscar).
- ImplementaĂ§Ă”es especĂ­ficas incluem **FilmeDAO**, **SalaDAO**, **SessaoDAO**, **IngressoDAO** e **VendaDAO**, que comunicam diretamente com o PostgreSQL atravĂ©s do `db_config.py`.

### Controller (Fluxo de Dados)
- Atuam como intermediĂĄrios entre a View e o DAO. Eles recebem os dados brutos da interface, realizam validaĂ§Ă”es, convertem tipos, aplicam regras de negĂłcio e instanciam as classes do Model antes de enviĂĄ-las ao banco de dados.

### View (Interface GrĂĄfica)
- Telas desenvolvidas em **Tkinter**.
- Divididas em Ărea Administrativa (CRUD de catĂĄlogos e consulta de histĂłrico de vendas) e Ărea do Cliente (Jornada interativa de compra de ingresso).

## PadrĂ”es de Design Adicionais

### PadrĂŁo Factory
O padrĂŁo Factory Ă© utilizado no `IngressoController` para instanciar diferentes tipos de ingressos (comum, meia, VIP) a partir da string selecionada pelo cliente na View.

### PadrĂŁo Strategy
O padrĂŁo Strategy Ă© utilizado no `VendaController` para definir diferentes mĂ©todos de pagamento, recebendo a string da View e instanciando o mĂ©todo correto para realizar a transaĂ§ĂŁo antes de salvĂĄ-la no banco de dados.

# Estrutura das Pastas e Pilares de POO
O sistema foi desenvolvido aplicando os pilares da POO (AbstraĂ§ĂŁo, Encapsulamento, HeranĂ§a e Polimorfismo), e tambĂ©m uma estrutura de diretĂłrios:

* **/model:** Encapsulamento de atributos via properties e mĂ©todos de classe com heranĂ§a (Ingresso e Pagamento) e polimorfismo (`calcular_preco()` e `processar_pagamento()`).
* **/dao:** AbstraĂ§ĂŁo e HeranĂ§a na implementaĂ§ĂŁo da comunicaĂ§ĂŁo SQL isolada por entidade.
* **/controller:** Encapsulamento da orquestraĂ§ĂŁo do sistema.
* **/view:** AbstraĂ§ĂŁo grĂĄfica atravĂ©s de heranĂ§as nativas do Tkinter (`tk.Toplevel`).

    def __init__(self):
        self.dao = FilmeDAO()

    def salvar(self, titulo: str, duracao_str: str, classificacao_str: str, status_str: str):
        try:
            duracao = int(duracao_str)
            if not titulo.strip():
                return False, "O tĂ­tulo nĂŁo pode ser vazio."
            
            classificacao = next(c for c in ClassificacaoIndicativa if c.value == classificacao_str)
            status = next(s for s in StatusFilme if s.value == status_str)
            
            filme = Filme(titulo=titulo, duracao=duracao, classificacao=classificacao, status=status)
            return self.dao.salvar(filme)
        except ValueError:
            return False, "DuraĂ§ĂŁo deve ser um nĂșmero inteiro vĂĄlido."
        except StopIteration:
            return False, "ClassificaĂ§ĂŁo ou Status invĂĄlido."

from datetime import datetime
from model.ingresso import IngressoFactory
from dao.ingresso_dao import IngressoDAO
from dao.sessao_dao import SessaoDAO

class IngressoController:
    def __init__(self):
        self.dao = IngressoDAO()
        self.sessao_dao = SessaoDAO()

    def gerar_ingresso(self, tipo: str, id_sessao: int, assento_str: str):
        try:
            sessao = self.sessao_dao.buscar_por_id(id_sessao)
            if not sessao:
                return False, "SessĂŁo nĂŁo encontrada."
            
            if sessao.data_hora <= datetime.now():
                return False, "NĂŁo Ă© possĂ­vel comprar ingressos para sessĂ”es que jĂĄ iniciaram ou ocorreram no passado."

            assento = int(assento_str)
            if assento <= 0 or assento > sessao.sala.capacidade:
                return False, f"Assento invĂĄlido. A sala suporta atĂ© {sessao.sala.capacidade} assentos."

            ingressos_vendidos = self.dao.listar()
            for ing in ingressos_vendidos:
                if ing.sessao.id_sessao == id_sessao and ing.assento == assento:
                    return False, "Assento jĂĄ ocupado nesta sessĂŁo."

            ingresso = IngressoFactory.criar_ingresso(tipo, sessao, assento)
            
            sucesso, msg = self.dao.salvar(ingresso)
            if sucesso:
                return True, ingresso
            return False, msg
            
        except ValueError as e:
            return False, str(e)
        except Exception as e:
            return False, f"Erro inesperado: {e}"

    def listar(self):
        return self.dao.listar()

        pass

    @abstractmethod

        pass

    @abstractmethod
    def listar(self):
        pass

    @abstractmethod
    def buscar_por_id(self, id_objeto: int):
        pass

from abc import ABC, abstractmethod
from model.sessao import Sessao

class Ingresso(ABC):
    def __init__(self, sessao: Sessao, assento: int, id_ingresso: int = None):
        self._id_ingresso = id_ingresso
        self.sessao = sessao
        self.assento = assento

    @property
    def id_ingresso(self) -> int:
        return self._id_ingresso

    @id_ingresso.setter
    def id_ingresso(self, valor: int):
        self._id_ingresso = valor

    @property
    def sessao(self) -> Sessao:
        return self._sessao

    @sessao.setter
    def sessao(self, valor: Sessao):
        if not isinstance(valor, Sessao):
            raise TypeError("O objeto deve ser do tipo Sessao.")
        self._sessao = valor

    @property
    def assento(self) -> int:
        return self._assento

    @assento.setter
    def assento(self, valor: int):
        if valor <= 0:
            raise ValueError("O assento deve ser um nĂșmero positivo.")
        self._assento = valor

    @abstractmethod
    def calcular_preco(self) -> float:
        pass

    @abstractmethod
    def get_tipo(self) -> str:
        pass


class IngressoComum(Ingresso):
    def calcular_preco(self) -> float:
        return float(self.sessao.preco_base)

    def get_tipo(self) -> str:
        return "Comum"


class IngressoMeia(Ingresso):
    def calcular_preco(self) -> float:
        return float(self.sessao.preco_base) * 0.5

    def get_tipo(self) -> str:
        return "Meia Entrada"


class IngressoVip(Ingresso):
    def calcular_preco(self) -> float:
        return float(self.sessao.preco_base) * 1.5

    def get_tipo(self) -> str:
        return "VIP"


class IngressoFactory:
    @staticmethod
    def criar_ingresso(tipo: str, sessao: Sessao, assento: int) -> Ingresso:
        tipo_formatado = tipo.upper().strip()
        
        if tipo_formatado == "COMUM":
            return IngressoComum(sessao, assento)
            
        elif tipo_formatado in ["MEIA", "MEIA ENTRADA", "MEIA-ENTRADA"]:
            return IngressoMeia(sessao, assento)
            
        elif tipo_formatado == "VIP":
            return IngressoVip(sessao, assento)
            
        else:
            raise ValueError(f"Tipo de ingresso invĂĄlido: '{tipo}'")

    id_sessao serial primary key,

    id_sala integer not null,
    data_hora timestamp not null,
    preco_base numeric(10, 2) not null check (preco_base >= 0),
    status varchar(50) not null,
    constraint fk_sessao_filme foreign key (id_filme) references tb_filmes (id_filme) on delete cascade,
    constraint fk_sessao_sala foreign key (id_sala) references tb_salas (id_sala) on delete cascade
);

create table if not exists tb_ingressos(
    id_ingresso serial primary key,
    id_sessao integer not null,
    assento integer not null check (assento > 0),
    tipo varchar(50) not null,
    constraint fk_ingresso_sessao foreign key (id_sessao) references tb_sessoes (id_sessao) on delete cascade,
    constraint uq_sessao_assento unique (id_sessao, assento) -- garante que um assento nĂŁo seja vendido duas vezes na mesma sessĂŁo
